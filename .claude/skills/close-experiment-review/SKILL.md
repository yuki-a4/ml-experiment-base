# 実験クローズ前の敵対的レビュー

実験を回した本人には、サンクコスト（かけた時間を無駄にしたくない）と確証バイアス（自分の結論を支持する証拠ばかり見る）がかかる。自分では「やり尽くした」と思っていても、少数試行からの早計な結論・要因分解の欠落・過度な一般化は、**KNOWLEDGE.md や EXPERIMENT.md に残った瞬間に後続実験の前提となり、誤りを伝播させる**。だからクローズの直前に、実験に関与していない独立のレビュアーが敵対的に審査する。

このレビューは自分（メインエージェント）ではやらない。自分で書いた REPORT を自分で審査すると必ず甘くなる — **必ずサブエージェントに委譲する**。

## いつ呼ぶか

実験ライフサイクルの中では、**REPORT.md を書き終え、EXPERIMENT.md / KNOWLEDGE.md への追記（または追記案）が揃った後、Issue をクローズする前**。

- 対象は種別を問わない（実験 / 分析(EDA) / 調査）。分析・調査も「結論」を出す以上、審査対象になる
- GitHub 連携が無い環境では「クローズ」= 記録を確定して実験を完了扱いにすること。その場合も同じ審査を通す
- ネガティブ結果で撤退するときこそ呼ぶ（撤退にも機序が要る）

## 手順

### 1. 材料を揃える

レビュアーに渡す情報をまとめる:

- リポジトリルートと対象実験のパス（`exp/expNNNN`）
- Issue 番号・本文・これまでのコメント（GitHub 連携時。レビュアーは gh を使えない前提で、本文をプロンプトに貼る）
- EXPERIMENT.md / KNOWLEDGE.md への追記内容（追記済みならその旨、案の段階なら案文）
- 審査ラウンド番号（初回は 1）

REPORT.md が未完成のまま呼ばない（審査対象が確定してから）。

### 2. 敵対レビュアーを起動する

レビュアーの定義はこのスキルと同じリポジトリの `.claude/agents/experiment-adversary.md` にある（このスキルのベースディレクトリから `../../agents/experiment-adversary.md`）。

- Agent tool の `subagent_type` に `experiment-adversary` が利用可能ならそれで起動する
- 利用できない場合は、定義ファイルを Read し、その本文を審査指示としてプロンプトの先頭に貼った上で `general-purpose` サブエージェントを起動する

いずれの場合も、プロンプトに手順 1 の材料をすべて含め、**結果を待ってから**次へ進む（バックグラウンドにしない）。

### 3. 判定に従う

レビュアーは `APPROVE / CONDITIONAL / REJECT` のいずれかを返す:

- **APPROVE** → クローズ処理へ（手順 4）
- **CONDITIONAL** → 非ブロッキング提案 [S] を処理してからクローズ: 提案ごとに既存 Issue との重複を確認して新 Issue を起票（または REPORT の記載修正）。その後クローズ処理へ
- **REJECT** → **クローズ禁止**。ブロッキング指摘 [B] を 1 件ずつ消化する:
  - (a) 指摘された追加検証を実施し、結果を REPORT.md に反映する（結論が変わったら EXPERIMENT.md / KNOWLEDGE.md も直す）
  - (b) 指摘が誤解や既検証事項に基づくなら、機序と証拠を添えた反論をまとめる（「そう思う」だけの反論は不可）
  - すべての [B] に対応したら、ラウンド番号を +1 し、**前回の指摘と各対応**を材料に加えて手順 2 から再審査する

**ラウンド上限は 3**。3 ラウンドで合意に至らない場合は、対立点（レビュアーの指摘・自分の対応・埋まらない差分）を整理してユーザーにエスカレーションし、判断を仰ぐ。上限があるのはレビューの無限ループを防ぐためであって、「3 回粘れば通れる」という意味ではない — 対応せずに同じ反論を繰り返さないこと。

### 4. 審査結果を記録してクローズする

- REPORT.md の末尾に `## クローズレビュー` 節を追加し、ラウンド数・最終判定・主要な指摘と対応を短く記録する（レビュアーの記名も転記）
- GitHub 連携時は、Issue のクローズコメントに審査結果（判定と要点）を含める
- [S] 由来の新 Issue を起票した場合は、そのリンクも残す

## やってはいけないこと

- REJECT のままクローズする / レビューをスキップして `gh issue close` する
- レビュアーの指摘を、検証も反論もせずに「対応済み」として再審査に出す
- 自分でレビューを代行して APPROVE 扱いにする（独立性が失われ、このゲートの意味がなくなる）
