# handoff/ の運用ルール（セッション引継ぎ）

ユーザーから「セッションを切り替えるので引継ぎプロンプトを作って」という趣旨の指示があったら、次セッションが**経緯と意図を失わずに**作業を再開できる引継ぎドキュメントを `handoff/` に作成する。

## ファイルの置き方・命名

- 置き場所はリポジトリ直下の `handoff/`。1 引継ぎ = 1 ファイル。
- ファイル名は **どの実験番号からの引継ぎかがわかる**ように `handoff_expNNNN_YYYYMMDD.md` とする（expNNNN = 引継ぎ時点で作業中／最新の実験番号。例: `handoff_exp0023_20260718.md`）。同日に複数作る場合は `_2` などの接尾辞を付ける。
- 実験をまたがない引継ぎ（セットアップ作業中など）は expNNNN の代わりに一言 slug でよい（例: `handoff_setup_20260718.md`）。
- 作成したら**コミット・push する**（次セッションが `git pull` で受け取れるように）。作成後、ユーザーにはファイルパスを伝え、次セッションでは「`handoff/<ファイル名>` を読んで再開して」とだけ指示すればよい状態にする。

## 書く内容（スコアや事実の羅列でなく「意図・経緯」を伝える）

冒頭に「このファイルは次セッションへの引継ぎです。まずこれを読んでから作業を再開してください」という一文を置き、以下を書く:

1. **現在地**: 作業中の expNNNN・対応する Issue 番号・実験ライフサイクルのどの段階まで進んだか（Issue 作成済み / 実験中 / REPORT 未記入 など）。
2. **経緯と意図**: なぜこの実験・作業をしているのか。どの仮説の流れの上にあり、直前までに何を試し、何が支持/棄却されたか。**次セッションが「なぜ」を再発掘しなくて済むように書く**。
3. **未完了タスクと次アクション**: 途中で止まっている作業、次にやるつもりだったこと（優先順位付き）。実行中・実行予定の長時間ジョブがあればコマンドとログの場所。
4. **注意点・ハマりどころ**: このセッションで踏んだ落とし穴、暫定的な判断（後で見直すべき点）、触ってはいけないもの。
5. **参照すべきファイル**: 関連する REPORT.md・Issue・survey/・出力物（OOF・図など）へのパス／リンク。
6. **記録者と日付**: 記録者（LLM なら自身のモデル名）と作成日を末尾に明記する。

## 住み分け

- **REPORT.md / KNOWLEDGE.md との違い**: それらは「確定した結果・知見」の記録。handoff/ は**未確定の途中状態と意図**を運ぶためのもの。確定した内容は handoff に書かず、通常どおり REPORT / EXPERIMENT / KNOWLEDGE に記録してからリンクする。
- 引継ぎ先セッションで役目を終えたファイルも**削除しない**（経緯の履歴として残す）。ただし内容が古いことは日付で判断する。
